# 법카자동화 Outlook 이메일 기능 설계 및 방식 비교 가이드

작성일: 2026-03-19  
대상 프로젝트: CARD_AUTO / 법카자동화  
목적: Outlook 메일 발송 기능을 어떤 방식으로 구현할지 결정할 수 있도록, **문서 주체를 `선임` / `GPT`로 명확히 구분**하여 두 가이드를 비교·정리한다.

---

## 1. 한눈에 보는 결론

현재 선택지는 크게 두 가지다.

1. **선임 방식**  
   - 성격: **사용자 로그인 기반(delegated permission) + public client + device code / native client 흐름**
   - 장점: 비교적 빨리 붙이기 쉽고, 관리자 협조가 적을 수도 있다.
   - 단점: 사람의 로그인/동의가 전제되고, 서버가 완전 무인 배치로 돌기에는 불안정하다.

2. **GPT 방식**  
   - 성격: **서버 자동발송 기반(app-only / client credentials) + 전용 발신 메일박스 + Microsoft Graph**
   - 장점: 법카자동화처럼 **시스템이 결과파일을 사용자에게 자동 발송**하는 시나리오에 더 적합하다.
   - 단점: 초기 Entra/Exchange 관리자 작업이 더 필요하고, 1차 세팅 난이도가 더 높다.

### 최종 추천

- **운영 목표가 “웹에서 버튼 누르면 시스템이 사용자들에게 결과파일을 자동 발송”이라면**  
  **GPT 가이드의 app-only 방식이 더 적합**하다.
- **운영 환경 세팅 권한이 없고, 빠르게 POC만 붙여야 하며, 로그인 1회/주기적 재인증을 감수할 수 있다면**  
  **선임 가이드 방식이 빠른 출발점**이 될 수 있다.

즉,

- **단기 POC / 권한 제약 큼** → 선임 방식 검토
- **실운영 / 배치 자동화 / 안정성 중시** → GPT 방식 권장

## 1-1. 최종 결론안

### 결론 한 줄
- **최종 운영안은 `GPT 가이드 방식`으로 채택**
- **선임 방식은 POC 또는 관리자 협조 전 임시 단계로만 활용**

### 문서 주체별 판단

#### 선임
- 핵심 방향: **delegated permission + public client + 사용자 로그인 기반 발송**
- 가장 잘 맞는 상황: **빠른 동작 검증, 소규모 수동 발송, 관리자 협조가 아직 어려운 상황**
- 장점 요약: **구현 진입이 빠르고, 실제로 당장 붙여보기 쉽다**
- 한계 요약: **사용자 로그인과 토큰 캐시에 의존하므로 서버 자동화/운영 안정성 측면에서 한계가 있다**

#### GPT
- 핵심 방향: **app-only + 전용 발신 메일박스 + 서버 워커 기반 자동 발송**
- 가장 잘 맞는 상황: **법카자동화 웹서비스의 실운영, 이력 관리, 실패 재시도, 시스템 발신자 일원화**
- 장점 요약: **무인 자동화, 운영 통제, 감사 로그, 재시도 구조에 유리하다**
- 한계 요약: **초기 Entra/Exchange 관리자 설정이 더 필요하다**

### 최종 의사결정안
1. **운영 목표가 명확하면 `GPT 가이드 방식`으로 바로 간다.**
2. **관리자 협조가 당장 어렵고 이번 주 안에 POC가 필요하면 `선임 방식`으로 먼저 붙인다.**
3. 단, 이 경우에도 **이메일 매핑/미리보기/배치/이력/재시도 구조는 GPT 가이드 기준으로 공통 설계**하고, **인증 모듈만 교체 가능하게 구현**한다.

### 추천 우선순위
- **1순위:** GPT 가이드 방식으로 최종 운영 설계 확정
- **2순위:** 필요 시 선임 방식을 임시 POC로 사용
- **3순위:** 선임 방식을 장기 운영안으로 고정하는 선택은 비권장

## 2. Microsoft 공식 기준으로 본 핵심 차이

Microsoft Graph는 **delegated access** 와 **app-only access** 두 가지를 지원한다. delegated는 로그인한 사용자를 대신해 동작하고, app-only는 로그인한 사용자 없이 앱 자체 권한으로 동작한다. 따라서 “누가 메일을 보내는지”, “사용자 로그인이 필요한지”, “서버 자동화에 적합한지”가 이 둘의 가장 큰 차이점이다.  
참고: Microsoft Learn - Graph permissions overview, sendMail API, device code flow, Exchange App RBAC

## 3. 선임 가이드 방식 요약

선임이 공유한 문서의 핵심은 아래와 같다.

### 3.1 인증/권한 구조
- Azure AD(현 Microsoft Entra) 앱 등록
- **Delegated permission** 으로 `Mail.Send` 추가
- `User.Read` 포함
- 리디렉션 URI에 native client / desktop 성격의 URI 추가
- **공용 클라이언트 흐름 허용**
- 최초 1회 사용자가 브라우저에서 로그인/인증
- 토큰을 파일(`o365_token.json`)에 저장 후 재사용

### 3.2 구현 흐름 추정
1. 사용자가 메일 발송 버튼 클릭
2. 최초 1회면 device code 또는 native public client 로그인 유도
3. 로그인한 사용자의 delegated token 획득
4. 해당 사용자 컨텍스트로 Graph `sendMail` 호출
5. 로컬/컨테이너 볼륨에 토큰 캐시 저장

### 3.3 이 방식이 잘 맞는 경우
- CLI 도구
- 개인/운영자 1명이 수동 실행하는 내부 툴
- 서버 무인 배치가 아니라 “사람이 실행해서 보내는 도구”
- 관리자 권한 협조가 어려워 application permission 구성이 힘든 경우

---

## 4. GPT 가이드 방식 요약

### 4.1 인증/권한 구조
- Microsoft Entra 앱 등록
- **Application permission** 으로 `Mail.Send`
- 서버는 **client credentials** 로 토큰 발급
- 사용자 로그인 불필요
- 발신자는 **전용 시스템 메일박스**(예: `card-auto@pine-partners.com`)로 고정
- Exchange Online **App RBAC** 로 앱이 접근 가능한 메일박스를 제한

### 4.2 구현 흐름
1. 사용자가 웹에서 결과파일 생성
2. 발송 미리보기 확인 후 “발송 요청” 클릭
3. 시스템이 발송 배치/항목을 DB에 저장
4. 별도 워커가 Graph 토큰 발급
5. `/users/{sender_upn}/sendMail` 호출로 자동 발송
6. 성공/실패를 DB에 기록하고 재시도 관리

### 4.3 이 방식이 잘 맞는 경우
- 웹서비스/백엔드 서버
- 결과파일을 사용자별로 자동 배포하는 배치 업무
- 운영자 로그인 없이 시스템이 스스로 메일 전송해야 하는 구조
- 이력/감사/재시도/안정성이 중요한 업무

## 6. 선임 가이드 방식의 장점

### 6.1 빠르게 붙이기 좋다
앱 등록 후 delegated `Mail.Send` 와 public client 흐름을 구성하면, 비교적 적은 구조 변경으로 발송을 붙일 수 있다. device code flow는 브라우저가 없는 환경에서도 별도 디바이스에서 로그인할 수 있도록 설계된 표준 흐름이다.

### 6.2 관리자 권한 의존도가 상대적으로 낮을 수 있다
application permission + App RBAC 구조보다 조직 관리자 협조가 덜 필요할 수 있다. 팀 내에서 “먼저 한번 붙여보자”는 목적에는 현실적이다.

### 6.3 운영자 1인 수동 발송 툴에는 잘 맞는다
예를 들어 담당자가 직접 웹에서 발송 버튼을 누르고, 자기 계정으로 로그인한 뒤 발송하는 구조라면 충분히 쓸 수 있다.

---

## 7. 선임 가이드 방식의 단점

### 7.1 서버 자동화에는 구조적으로 덜 맞다
delegated access는 본질적으로 **로그인한 사용자 대신** 동작한다. 즉, “시스템이 사람 없이 자동으로 꾸준히 보내는” 구조와는 철학이 다르다.

### 7.2 사용자 계정 상태에 영향을 많이 받는다
비밀번호 변경, MFA 정책 변경, 계정 잠금, 사용자 퇴사, 토큰 폐기 등의 영향을 받을 수 있다. 자동발송 시스템에서는 이런 운영 리스크가 커진다.

### 7.3 발신 주체가 불명확해질 수 있다
누구 계정으로 로그인했는지에 따라 발신자가 달라질 수 있다. 법카자동화처럼 **공식 시스템 메일**로 보내는 성격의 업무에는 일관성이 떨어진다.

### 7.4 토큰 파일 운영이 깔끔하지 않을 수 있다
`/app/data/o365_token.json` 같은 로컬 토큰 캐시 의존은 단일 컨테이너/단일 운영자 환경에서는 가능하지만, 다중 인스턴스·서버 이전·권한 분리 환경에서는 관리가 복잡해질 수 있다.

### 7.5 설계 문서 일부가 현재 시나리오와 안 맞을 수 있다
선임 문서에는 리디렉션 URI, public client, device-like 흐름이 강조되어 있는데, 이것은 **“사람이 로그인하는 앱”** 에 맞는 요소다. 지금 목표가 **백엔드 자동 발송** 이라면 필수 요건이 아니다.

## 8. GPT 가이드 방식의 장점

### 8.1 현재 요구사항과 가장 잘 맞다
요구사항은 사실상 **“결과파일을 사용자 이메일로 자동 전송하는 서버 기능”** 이다. 이건 app-only가 더 자연스럽다.

### 8.2 발신자를 전용 시스템 메일박스로 고정할 수 있다
예를 들어 `card-auto@pine-partners.com` 같은 계정으로 일관되게 발송하면, 사용자는 발신 주체를 명확히 인지할 수 있고 운영/문의 체계도 단순해진다.

### 8.3 이력·감사·재시도 구조와 잘 결합된다
배치 발송, 발송 큐, 실패 건 재시도, 발송 이력 저장 등 업무 시스템에 필요한 운영 기능과 궁합이 좋다.

### 8.4 사용자 로그인 의존이 없다
버튼만 누르면 서버가 처리할 수 있으므로, “로그인 유도 → 코드 입력 → 인증 완료” 같은 사용자 경험을 없앨 수 있다.

### 8.5 보안 통제가 명확하다
Graph `Mail.Send` application permission에 더해 Exchange Online **App RBAC** 를 적용하면, 앱이 접근할 수 있는 메일박스를 특정 발신 메일박스로 제한할 수 있다.

## 9. GPT 가이드 방식의 단점

### 9.1 초기 세팅이 더 어렵다
Entra 앱 등록, application permission, admin consent, Exchange Online App RBAC, 발신용 사서함 준비가 필요하다.

### 9.2 관리자 협조가 사실상 필수다
개발자가 혼자 로컬에서 바로 끝내기 어려울 수 있다. 특히 운영 테넌트에서는 M365/Exchange 관리자 지원이 필요하다.

### 9.3 작은 PoC에는 과해 보일 수 있다
일단 1~2명이 수동 발송만 해보는 단계라면 delegated 방식보다 무겁게 느껴질 수 있다.

## 10. 실제 법카자동화 요구사항 기준 비교

### 10.1 현재 업무 특징
- 결과파일을 **여러 사용자에게 반복적으로 발송**할 가능성이 높음
- 카드 사용자명과 이메일 매핑이 필요함
- 발송 성공/실패 이력이 중요함
- 추후 재시도와 감사 로그가 필요할 가능성이 큼
- 발송자는 개인이 아니라 **시스템** 이어야 자연스러움

이 조건을 보면, **단일 운영자 로그인 기반**보다 **시스템 전용 발신자 + 서버 워커** 구조가 훨씬 자연스럽다.

### 10.2 따라서 더 적합한 방식
운영 관점 기준으로는 **GPT 가이드 방식이 우세**하다.

다만, 아래 조건이면 선임 방식도 쓸 수 있다.

- 지금 당장 관리팀 협조가 어렵다.
- 먼저 “메일이 나가는지”만 빠르게 검증하고 싶다.
- 담당자 1명이 사실상 수동 실행할 계획이다.
- 운영 안정성보다 단기 개발 속도가 더 중요하다.

## 11. 추천 의사결정안

### 안 A. 바로 운영형으로 간다
**추천 대상:** 실제 서비스 반영이 목적이고, 관리자 협조를 받을 수 있는 경우

구성:
- Microsoft Graph
- Application permission (`Mail.Send`)
- Client credentials
- 전용 발신 메일박스
- App RBAC
- DB 기반 발송 배치/항목/재시도

**장점**
- 재작업이 적다
- 운영에 바로 가까운 구조
- 추후 확장 쉬움

**단점**
- 시작이 느릴 수 있다

### 안 B. 선임 방식으로 먼저 POC 후, 운영형으로 전환한다
**추천 대상:** 지금 바로 동작 확인이 우선이고, 관리자 작업이 아직 어려운 경우

1단계:
- delegated + device/public client로 우선 발송 확인
- 화면/파일/발송 이력 UI 먼저 완성

2단계:
- app-only + 전용 발신 메일박스로 인증 구조 교체

**장점**
- 빨리 움직일 수 있다
- 프론트/업무 흐름을 먼저 검증 가능

**단점**
- 인증 로직을 나중에 갈아엎을 가능성이 높다

### 안 C. 선임 방식 그대로 운영한다
**추천 대상:** 사실상 없음(단, 매우 작은 내부 수동 툴 제외)

지금 프로젝트가 “법카자동화 웹페이지”라는 점을 감안하면, 장기 운영 구조로는 권장하지 않는다.

## 12. GPT 최종 권고

### 권고 1
**최종 운영 목표는 app-only 방식으로 잡는 것이 맞다.**

이유:
- 시스템 자동화에 적합
- 발신자 일관성 확보
- 재시도/로그/배치 구조에 적합
- 사용자 로그인 의존 제거

### 권고 2
다만 현실적으로 관리자 세팅이 당장 어렵다면,
**선임 방식으로 빠르게 “발송 기능 UI/업무 흐름”만 먼저 검증**하는 것은 가능하다.

단, 이 경우에도 아래는 꼭 분리해서 개발해야 한다.
- 이메일 매핑 테이블
- 발송 배치/항목 테이블
- 발송 미리보기
- 실패/재시도 UI
- 발송 서비스 인터페이스

즉, **인증 구현만 교체 가능하게 추상화** 해두면 된다.

## 13. 선임 가이드와 내 가이드를 섞어 쓰는 현실적인 절충안

### 13.1 공통으로 바로 가져갈 수 있는 부분
두 방식과 상관없이 아래는 동일하게 필요하다.

- 사용자명 ↔ 이메일 매핑 관리
- 발송 전 미리보기
- 첨부파일 크기 검증
- 발송 이력 저장
- 실패 건 재시도
- 결과파일 경로 검증
- 템플릿 제목/본문

### 13.2 교체 가능하게 만들어야 할 부분
- 토큰 발급부
- Graph 메일 발송부
- 발신자 식별 방식

즉, 코드 구조는 아래처럼 가는 것이 좋다.

- `mail_graph_service.py`
  - `send_mail(...)`
- 구현 A: delegated/device-flow 버전
- 구현 B: app-only/client-credentials 버전

이렇게 하면 선임 방식으로 먼저 붙였다가 나중에 app-only로 갈아타기 쉽다.

## 14. 선임 가이드 문서에서 특히 눈여겨볼 점

### 14.1 장점
- 실제로 이미 동작하는 사례일 가능성이 높다.
- Docker 환경까지 고려한 토큰 저장 흐름이 들어 있다.
- 개발자가 따라 하기 쉬운 단계별 문서다.

### 14.2 주의점
- 현재 문서 마지막에 **앱 이름, 클라이언트 ID, 테넌트 ID, 발신자 이메일** 이 그대로 적혀 있다.
- 클라이언트 ID와 테넌트 ID 자체가 비밀번호는 아니지만, 내부 문서를 외부/광범위하게 공유할 때는 굳이 노출하지 않는 편이 좋다.
- 특히 운영 문서에는 **실제 운영값 예시를 마스킹** 해서 남기는 습관이 좋다.

## 15. 지금 당장 무엇을 확인하면 되는가

의사결정 전에 아래 6가지를 먼저 확인하면 된다.

1. **이 기능이 진짜 무인 자동발송인가, 아니면 담당자가 수동으로 보내는가?**  
   - 무인 자동발송이면 app-only 쪽이 더 적합

2. **M365 / Entra / Exchange 관리자 협조를 바로 받을 수 있는가?**  
   - 가능하면 app-only 추천
   - 불가능하면 delegated POC부터 가능

3. **발신자는 누구여야 하는가?**  
   - 개인 계정이면 delegated도 가능
   - 시스템 대표 메일이면 app-only가 자연스러움

4. **운영상 재인증/계정 이슈를 감수할 수 있는가?**  
   - 감수 어렵다면 app-only

5. **장기적으로 발송 이력/감사/재시도/배치 기능이 필요한가?**  
   - 필요하면 app-only가 더 잘 맞음

6. **지금 목표가 POC인가, 운영 반영인가?**  
   - POC면 선임 방식도 충분히 검토 가능
   - 운영이면 app-only 우선

## 16. 최종 추천 시나리오

### 시나리오 A: 이번 달 안에 운영 반영해야 함
- **GPT 방식(app-only)** 으로 가는 것이 좋다.
- 처음부터 발송 배치/항목/이력/재시도까지 설계한다.

### 시나리오 B: 일단 메일 발송 기능이 되는지만 빨리 보여줘야 함
- **선임 방식(delegated/device/public client)** 으로 POC 가능
- 단, 메일 발송 서비스 인터페이스는 교체 가능하게 만든다.

### 시나리오 C: 관리자 설정은 늦고, 화면 개발은 먼저 해야 함
- 인증부를 mock 또는 인터페이스로 두고
- 이메일 매핑 / 미리보기 / 발송 이력 / 실패 처리 화면을 먼저 만든다.

## 17. GPT의 최종 판단

현 시점에서 **법카자동화의 “정답 구조”는 app-only + 전용 발신 메일박스 방식**이다.  
다만 개발 일정과 관리자 협조 여부 때문에 **선임 방식이 “빠른 중간 단계”** 가 될 수는 있다.

따라서 가장 현실적인 전략은 아래 둘 중 하나다.

### 가장 추천
- **운영 목표 확정 + 관리자 협조 가능**  
  → 처음부터 app-only로 간다.

### 차선책
- **지금 당장 POC 필요 + 관리자 협조 불확실**  
  → 선임 방식으로 먼저 붙이되, 서비스 레이어를 추상화해서 나중에 app-only로 전환한다.

## 19. 참고용 결론 문장

팀 내부 의사결정 문구로는 아래처럼 정리하면 된다.

> 선임이 공유한 방식은 delegated/public-client 기반으로 빠르게 메일 발송 POC를 구현하기에 유리하나, 사용자 로그인과 토큰 캐시에 의존하므로 법카자동화 웹서비스의 장기 운영 구조로는 한계가 있다. 반면 app-only + 전용 발신 메일박스 방식은 초기 관리자 작업이 더 필요하지만, 시스템 자동발송, 운영 안정성, 이력 관리, 보안 통제 측면에서 본 프로젝트 요구사항에 더 적합하다. 따라서 최종 운영안은 app-only 구조를 목표로 하고, 필요 시 delegated 방식은 임시 POC 수단으로만 활용하는 것이 바람직하다.
